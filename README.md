# TUMO Astana — система приёма, очереди и кураторства (прототип)

Рабочий прототип, демонстрирующий ключевую автоматизацию для приёмной кампании
и работы кураторов. Все данные — тестовые (мок), в памяти приложения.
Реальной интеграции с T360 и реального распознавания лиц здесь нет — это
сознательное решение на этапе прототипа (см. раздел "Что дальше").

## Запуск локально

Нужен установленный Node.js (версия 18 или новее).

```bash
npm install
npm run dev
```

Откроется адрес вида `http://localhost:5173` — там работает всё приложение.

## Что внутри

- **Главная** (`/`) — выбор роли, живой индикатор заполненности текущего потока.
- **Анкета для родителей** (`/apply`) — подача заявки. Родитель выбирает
  ровно 4 направления из 11 (чекбоксы с live-счётчиком «выбрано X из 4»).
  Если в потоке есть свободное место, заявка сразу получает статус
  «принята». Если мест нет — заявка автоматически встаёт в очередь со своей
  позицией. Лимит мест в потоке общий на ребёнка и не зависит от направлений.
- **Статус заявки** (`/status`) — родитель вводит номер заявки и видит статус
  и позицию в очереди.
- **Кабинет куратора** (`/curator`, демо-логин `a.bekova` / `demo123`) —
  список закреплённых учеников, отметка посещаемости по дням, заметки
  о прогрессе.
- **Админ-панель** (`/admin`) — изменение лимита мест в потоке. При
  увеличении лимита система автоматически переводит из очереди в «принятые»
  нужное число заявок по порядку. При отчислении принятого ученика место
  освобождается и тоже подтягивается следующий из очереди.
- **Концепт турникета** (`/face-concept`) — макет сценария «распознавание
  лица → открытие турникета» на вымышленных тестовых профилях, без камеры
  и без реальных данных детей. Сделан, чтобы показать идею на презентации,
  не запуская реальный сбор биометрии.

## Архитектура данных

Файл `src/data/store.js` — это вся бизнес-логика и "база данных" в одном
месте: потоки (`streams`), заявки (`applications`), кураторы (`curators`),
ученики (`students`), посещаемость (`attendance`), заметки о прогрессе
(`progressNotes`).

Это сделано намеренно как единая точка, которую нужно заменить при переходе
от прототипа к реальному внедрению: все функции (`submitApplication`,
`setStreamCapacity`, `markAttendance` и т.д.) можно один за одним переписать
так, чтобы они обращались не к памяти, а к настоящей базе данных
(например, Supabase/Postgres) или к API. Остальной код (формы, страницы,
дизайн) можно не трогать.

## Что дальше — для обсуждения с руководством

Этот прототип закрывает: автоматизацию очереди и лимитов, цифровой приём
заявок, портал кураторов с посещаемостью и прогрессом. Это можно показать
и обсуждать уже сейчас.

Три вещи требуют отдельного решения организации, а не просто доработки кода:

1. **Интеграция с T360.** Нужно узнать, есть ли у T360 официальный API для
   создания аккаунтов учеников программно. Без него автоматическое создание
   аккаунтов невозможно — это вопрос к поставщику платформы T360, а не
   к разработке.
2. **Сбор биометрии детей (распознавание лиц на турникете).** Технически
   реализуемо, но требует: официального решения руководства центра о сборе
   биометрических данных несовершеннолетних, согласованной с юристами формы
   согласия родителей, политики хранения и защиты данных, соответствующей
   требованиям законодательства Казахстана о персональных данных. Это
   институциональный вопрос, который должен быть решён до начала разработки
   этой части, а не после.
3. **Мобильное приложение.** Видео-инструкция «как пройти в кабинет» — это
   простая часть. Полноценное приложение для родителей/кураторов на
   Flutter — следующий этап после того, как веб-версия проверена в реальной
   работе.
